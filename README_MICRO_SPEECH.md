
# Micro Speech Training

This example shows how to train a less than 20 kB model that can recognize 2 keywords,
"yes" and "no", from speech data.

If the input does not belong to either categories, it is classified as "unknown"
and if the input is silent, it is classified as "silence".

You can retrain it to recognize any combination of words (2 or more) from this
list:

```
yes
no
up
down
left
right
on
off
stop
go
```

The scripts used in training the model have been sourced from the
[Simple Audio Recognition](https://www.tensorflow.org/tutorials/audio/simple_audio)
tutorial.

## Model Architecture

This is a simple model comprised of a Convolutional 2D layer, a Fully Connected
Layer or a MatMul Layer (output: logits) and a Softmax layer
(output: probabilities) as shown below. Refer to the [`tiny_conv`](https://github.com/tensorflow/tensorflow/blob/master/tensorflow/examples/speech_commands/models.py#L673)
model architecture.

[<img src="../images/micro_speech_quantized.png" width="900" alt="model architecture"/>](../images/micro_speech_quantized.png)

*This image was derived from visualizing the 'models/micro_speech_quantized.tflite' file in
[Netron](https://github.com/lutzroeder/netron)*

This doesn't produce a highly accurate model, but it's designed to be used as
the first stage of a pipeline, running on a low-energy piece of hardware that
can always be on, and then wake higher-power chips when a possible utterance has
been found, so that more accurate analysis can be done. Additionally, the model
takes in preprocessed speech input as a result of which we can leverage a
simpler model for inference results.

## Dataset

The Speech Commands Dataset. ([Download Link](https://storage.cloud.google.com/download.tensorflow.org/data/speech_commands_v0.02.tar.gz),
[Paper](https://arxiv.org/abs/1804.03209)) consists of over 105,000 WAVE audio
files of people saying thirty different words. This data was collected by
Google and released under a CC BY license. You can help improve it by
contributing five minutes of your own voice. The archive is over 2GB, so this
part may take a while, but you should see progress logs, and once it's been
downloaded you won't need to do this again.

## Preprocessing Speech Input

In this section we discuss spectrograms, the preprocessed speech input to the
model.

The model doesn't take in raw audio sample data, instead it works with
spectrograms which are two dimensional arrays that are made up of slices of
frequency information, each taken from a different time window.

The recipe for creating the spectrogram data is that each frequency slice is
created by running an FFT across a 30ms window of the audio sample data. The
input samples are treated as being between -1 and +1 as real values (encoded as
-32,768 and 32,767 in 16-bit signed integer samples).  The audio sampling window
stride is 20ms, thus every window overlaps by 10ms.

This results in an FFT with 257 entries. Every sequence of approximately six entries is
averaged together, giving a total of 40 frequency buckets in the slice.
The results are further processed by down-scaling, noise reduction, automatic
gain control, and a final down-scaling.

Each adjacent frequency entry is stored in ascending memory order (frequency
bucket 0 at data[0], bucket 1 at data[1], etc). The window for the frequency
analysis is then moved forward by 20ms, and the process repeated, storing the
results of the new frequency slice in the next memory row.
The training is configured for raw audio samples of 1000ms in length.
With a window size of 30ms and stride of 20ms, some 49 frequency slices can
be created from 1000ms of audio data.
Thus, the preprocessing produces a
single channel image that is 40 pixels wide, and 49 rows high.

## Firmware integration (this repository, MIK32 / MAX9814)

This section ties the [Preprocessing Speech Input](#preprocessing-speech-input) recipe above to on-device code and to TensorFlow Lite Micro.

### What the trained model actually expects

- **Input**: not raw ADC counts. A **spectrogram** tensor: **40** frequency bins × **49** time rows (**1960** values), produced from **16 kHz** mono audio using a **30 ms** analysis window and **20 ms** stride (10 ms overlap), with the FFT / binning / noise reduction / AGC pipeline described in this README (and in `tensorflow/lite/micro/examples/micro_speech/train/README.md`).
- **Encoding**: training treats waveform samples as **−1 … +1** in float, encoded as **int16 −32768 … 32767** before that DSP stack.

### MAX9814 + 12-bit ADC

- **MAX9814** gives an **AC‑coupled**, **biased** analog mic signal. Your ADC sees a **resting midpoint** (not necessarily exactly 2048). Use **`feature_extractor_set_cal()`** (`ml/feature_extractor.h`) to set **`adc_midpoint`** from silence, and **`mic_gain_q8`** so speech peaks use a reasonable fraction of int16 range without clipping.
- **Sample rate**: the framing constants (`FE_FRAME_LEN`, `FE_FRAME_HOP`) assume **16 000 calls per second** into **`feature_extractor_push_adc`** or **`feature_extractor_push_pcm16`**. If your loop is slower, you must use a **timer interrupt**, **DMA**, or **oversampling + decimation** to hit 16 kHz, or retrain with a different rate and change the constants.

### “Proper” spectrograms on MCU (two realistic options)

1. **Official path (recommended for bit‑exact match)**  
   Run TensorFlow’s **`audio_preprocessor_int8.tflite`** as a **second** `MicroInterpreter`: each stride, feed **480 int16** samples, read **40 int8** bins, fill one row of the 49×40 buffer. Resolver ops are listed in `tensorflow/lite/micro/examples/micro_speech/micro_speech_test.cc` (`RegisterOps` for the preprocessor).  
   **RAM caveat:** the upstream example uses a **large** tensor arena for that subgraph; on **16 KiB RAM** you may need **one arena reused in two phases** (construct preprocessor → generate full 49×40 → destroy interpreter → construct keyword model) or a bigger MCU / external RAM.

2. **Port the training DSP in C**  
   Implement FFT + bin merge to 40 bands + same NR/AGC as training (heavy).

### Using `kissfft` from this project’s TFLM build

- **Sources**: `kissfft` is downloaded next to the TFLM Makefile (e.g. `third_party/tflite-micro/tensorflow/lite/micro/tools/make/downloads/kissfft/`). Wrappers live under `third_party/tflite-micro/signal/src/kiss_fft_wrappers/` (`kiss_fft_int16.cc` **includes** `kiss_fft.c` / `kiss_fftr.c` inside the `kiss_fft_fixed16` namespace).
- **Inside `libtensorflow-microlite.a`**: the build archives **`kiss_fft_int16.o`** (and float/int32 variants). You can call **`kiss_fft_fixed16::kiss_fftr_alloc` / `kiss_fftr`** by including **`signal/src/kiss_fft_wrappers/kiss_fft_int16.h`** and **not** compiling `kiss_fft_int16.cc` again (avoid duplicate symbols).
- **Linker behavior**: with **`--gc-sections`**, nothing from `kiss_fft_int16.o` is pulled in until **some translation unit** references those symbols. If you add your own FFT helper, the linker will take the object file from the `.a` as needed.
- **RAM on MIK32 (16 KiB)**: actually **linking** the fixed16 real FFT path pulls non‑trivial **`.text` / `.ram_text` / scratch**; our optional **`kiss_fft_link_test`** (`ml/kiss_fft_link_test.cc`) exceeded the chip’s RAM when forced into the image. It is gated by CMake **`ENABLE_KISS_FFT_LINK_TEST`** (default **OFF**). Turn it on only on a part with headroom, or keep FFT usage on the host / a bigger MCU.
- **How to try the link test**: `cmake -B build -DENABLE_KISS_FFT_LINK_TEST=ON` then build; inspect **`base_project.map`** for `kiss_fft_fixed16`.

The module **`ml/feature_extractor.c`** today keeps correct **timing and shape** but still fills rows with a **legacy energy proxy** (clearly marked in source). That is enough for plumbing tests, **not** for keyword accuracy until you replace the row generator with (1) or (2).

## Runtime RAM: what to measure and how

### Functions / data you already have

| Mechanism | What it tells you | Scope |
|-----------|-------------------|--------|
| **`MicroInterpreter::arena_used_bytes()`** | Bytes used inside the **tensor arena** after `AllocateTensors()` | TFLM scratch + tensor buffers only |
| **`tflm_arena_used_bytes()`** (`ml/tflm_wrapper.cc`) | Thin C wrapper around the above | Same |
| **Linker map / `.map` file** | **Static** `.bss`, `.data`, `.stack`, code in RAM (e.g. `.ram_text`) | Whole image; best view of **16 KiB** budget |
| **`_end` / `__BSS_END__`** (linker script) | End of static allocator region | Approximate heap start if you used heap (this project avoids heap for TFLM) |

**`arena_used_bytes()` does not include:** C stack, static globals outside the arena (e.g. `tensor_arena[]`, HAL buffers), or a second model’s arena if you add one.

### Suggested workflow

1. After a successful **`AllocateTensors()`**, log **`tflm_arena_used_bytes()`** and set **`kArenaSize`** to used + a few hundred bytes margin (`ml/tflm_wrapper.cc`).
2. Generate a **linker map** (add `-Wl,-Map=fw.map` to your link flags) and check **REGION_RAM** usage vs **stack** reservation in `hardware/mik32v2-shared/ldscripts/spifi.ld` / `sections.lds`.
3. Under **`release_with_logs`**, use **`MicroPrintf`** / your **`Mik32DebugLogWrite`** path for allocation errors (`AllocateTensors` failures).

## What we fixed in this project (short history)

1. **Toolchain / CMake** — C++ flags aligned with the RISC‑V core used by TFLM’s `mik32` makefile.
2. **`AllocateTensors` hard fault** — removed an illegal early call to **`PrepareNodeAndRegistrationDataFromFlatbuffer()`**; preparation must run only inside TFLM’s allocation flow.
3. **`MicroPrintf` / `debug_log`** — bare‑metal logging routed away from `vfprintf(stderr)`.
4. **Linker overlap** — stack vs `.bss`/`.ram_text` in 16 KiB; adjusted **`STACK_SIZE`** and removed duplicate **`MicroPrintf`** when linking **`release_with_logs`** microlite.
5. **UINT8 Softmax** — the keyword model used **uint8** activations on Softmax; micro’s reference Softmax only wired float/int8/int16. **Patches** in `third_party/tflite-micro/tensorflow/lite/micro/kernels/softmax.cc` and `softmax_common.cc` added the missing path (rebuild microlite with **`make -B`** or **`touch`** sources so **`Nothing to be done`** does not skip the fix).

### Avoiding repeat pain

- After editing TFLM sources, **`make -B ... microlite`** or **`touch`** the changed `.cc` files so the static library actually rebuilds.
- Pin **`third_party/tflite-micro`** (submodule or fork) whenever you carry **non‑upstream** kernel patches.
- Prefer **`arena_used_bytes()`** over guessing arena size on small RAM targets.
